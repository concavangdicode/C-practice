Question 1: Multimap

This question is worth 100% of the assignment mark.

A multimap is an Abstract Data Type where every value that we store is identified by a key. To add a new item to our data structure, we supply both a key and a value. To retrieve a value, we look it up using its key. A regular map has one value for each key; in a multimap, which we will build, a key may have multiple values.

An example of a multimap is a dictionary. The words are the keys, and each different definition for the word is a value. It can be considered a multimap rather than a regular map because there may be more than one definition for each word. For example, the Merriam-Webster dictionary definition of the noun "map" has three values for one key:

  1. a representation usually on a flat surface of the whole or a part of an area
  2. a representation of the celestial sphere or a part of it
  3. a diagram or other visual representation that shows the relative position of the parts of something

To which we would add:

  1. an ADT where stored values are accessed by providing an associated key

In our implementation, the keys will be strings, and values are a pair consisting of both an integer and a string together. Nether the numeric nor the string part of the value need to be unique for the key.

The list of keys will be stored in a partially-filled array, and the values for each key are stored using a linked list. Therefore, our data structure will be implemented as an array of linked lists. Both are dynamically allocated. The maximum size of the array (capacity) will be chosen when the multimap is created. The array will be ordered alphabetically (case insensitive) for quick lookup of keys using binary search. The values for each key will be ordered numerically (ascending) when they are retrieved.

Interface

The multimap is accessed with the following interface:

// These maximum string lengths include the \0.
// Your multimap should still work if these numbers change, though your
//  tests will be based on these numbers.
#define MAX_KEY_LENGTH 11
#define MAX_VALUE_LENGTH 21

typedef struct VALUE { int num; char str[MAX_VALUE_LENGTH]; } Value;
typedef struct MULTIMAP Multimap; // you need to define this yourself

// Create a new multimap with at most "max_keys" keys.
// Return NULL on error.
Multimap *mm_create(int max_keys);

// Insert a new value into the multimap for the given key.
// If the key already exists in the multimap, the value is added to that key.
// Return the number of values associated with this key after insertion.
int mm_insert_value(Multimap *mm, char *key, int value_num, char *value_str);

// Count the number of keys in the multimap.
int mm_count_keys(Multimap *mm);

// Count the number of values associated with the given key.
// Zero means the key is not found in the multimap.
int mm_count_values(Multimap *mm, char *key);

// Copy as many values as possible for the given key into the array.
// Values will be ordered (and copied) numerically.
// Note that while values[] contains copies of the structs, the str pointers
//  still point to memory that belongs to the Multimap. Do not free or modify!
// Return the number of values copied, at most max_values.
int mm_get_values(Multimap *mm, char *key, Value values[], int max_values);

// Remove the key and all its corresponding values from the multimap.
// Return the number of values that were removed with the key.
// Zero means the key is not found in the multimap.
int mm_remove_key(Multimap *mm, char *key);

// Print the contents of the multimap, neatly.
// Keys and values must be in the correct order.
// This will be helpful for debugging and manual testing.
void mm_print(Multimap *mm);

// Destroy a multimap, freeing all allocated memory.
// Return the total number of keys AND values that were freed.
int mm_destroy(Multimap *mm);

In all the functions that return an int, a return value of -1 indicates an error occurred.

You must also add your own tests. Organize them into functions, and ensure that you include the following types of tests:

  • typical cases
  • unusual cases, including empty cases
  • edge/boundary cases
  • invalid cases

You may want one or more functions for each type of test, or you may choose to organize them differently. Test functions may call other test functions. In your output, print a label for the type of test, as a group, to assist the marker in determining what kinds of tests you have for each.

There should be some tests that only pass (and are therefore only run) with assertions turned off. Wrap these tests in #ifdef NDEBUG and #endif pairs. Other tests will run regardless of whether the assertions are on or off.

Follow all good testing practices when you write your tests. Make sure that your tests test something: that is, they must do something and validate the results. Some functions, like mm_print, can't be tested with automated tests.

For your first attempt at writing tests, appropriately thorough testing should pass (likely much) more than 200 non-trivial verifications. This is a complicated data structure, there is much that can go wrong!

Unlike in assignment 2, the tests can verify specific outcomes. For example, use verify_int() rather than verify_non_negative() because you would know exactly what numeric value should be returned from a function. Unfortunately, this does not apply to pointers, where we can only check whether or not they are NULL. We can, however, check strings for exact matches.

Although the markers will run, read, and evaluate your tests, they will also run their own set of tests; this s similar to what they did in assignment 2, except this time you will not be given those tests in advance. If your own tests are good, you should have no problem passing the marker's tests. To ensure their tests run, do not change the interface or the given test harness. You can modify the main program as you like, since the markers will run their own replacement.
